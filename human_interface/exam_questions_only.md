### 問題G
ニールセンの10ヒューリスティックの「Visibility of system status」および「Error prevention」を説明するとともに、それに問題を持つインタフェースにはISO9241のユーザビリティ各項目にどのような問題が生じるか説明しなさい。

Visibility of system statusとは、現在のシステムがどのような状態化を示せということである。例えば何かを読み込んでいる際は砂時計を表示したり、progress barを表示することで裏で動いているかを表示するなどである。これがない場合は、ユーザーは実際に裏で実行されているのかわからず、再度実行してエラーが出ることでユーザの目的が達成できなくなったり、初めからやり直すことになり無駄な手順を踏むことになったり、現在何がおきているかわからずユーザーが不快感を感じることがある。
---

### 問題H
ニールセンの10ヒューリスティックの「Recognition rather than recall」および「User control and freedom」を説明するとともに、それに問題を持つインタフェースにはISO9241のユーザビリティ各項目にどのような問題が生じるか説明しなさい。

Recognition rather than recall
記憶させるのではなく、すぐに引き出せるようにするというような項目である。例えば、商品購入サイトでは購入ページでも何を購入したのかの個数や、細かな商品名などを確認できるようにして、ユーザの記憶の必要をなくす。問題があるサービスでは、カートを表示した際に個数や商品の細かい名前が表示されておらず、ユーザーが正しいものを選択できているか不安になり、購入できずゴールにたどり着けなくなったり、初めからやり直すことになり効率が悪くなったり、知りたい情報が得られずストレスが溜まり満足度が下がったりする。

User control and freedomについては undo, redoができるようにせよというものである。この点に問題があるサービスでは、undoができないことで最初からやり直すことになり、何度も同じ間違えをすることでゴールにたどり着けなくなったり、初めからやり直すことになり効率が悪くなったり、前に戻れないという不自由さから満足度が下がる可能性がある。

物理的制約、意味敵制約、文化的制約、論理的制約
ユーザモデル、デザインモデル、システムイメージ

### 問題I

発話プロトコル法とは、治験者に発話しつつプロダクトを使ってもらうという評価方法である。なにがしたいか、何を見ているか何を想像しているかなどを治験者に全て発話してもらうことで、どこで操作が詰まるのか、どこが目立つかなどを確認することができる。デメリットとしては、観察者側の労力が大きい点と、治験者が無口・おしゃべりによって評価が大きく変わる点と、すべての思考が言語化されるわけではないことである。

## 【分野8】アフォーダンス応用・ヒューリスティック評価法

### 問題O
「反アフォーダンス」とは何か説明しなさい。また、それが問題を引き起こした例を挙げなさい。

反アフォーダンス：逆に「できないこと・阻止すること」の性質

透明ガラスの例：
・「透かして見る」をアフォードする ← アフォーダンス
・物理的なモノが通り抜けるのを阻止する ← 反アフォーダンス

透明ガラスは「通り抜けを阻止する」反アフォーダンスを持つが、透明なので見えない。
→ 鳥が窓を通り抜けようとしてガラスに衝突する
→ 人がガラスドアに気づかず走り込んで怪我をする

ポイント：アフォーダンスも反アフォーダンスも、知覚できなければ機能しない。

---

### 問題P
「知覚されるアフォーダンス」とは何か、知覚されないアフォーダンスとの違いがわかるように説明しなさい。

アフォーダンスは人とモノの関係として常に存在する。でも気づかれないと使われない。知覚されて初めてユーザの行動につながる。

・ボタンの出っ張り → 「押せる」と知覚される ✅ 知覚されるアフォーダンス
・タッチスクリーンの表面 → 見た目では押せるかわからない ❌ 知覚されにくい

---

### 問題Q
ヒューリスティック評価法とはどのような評価手法か、発話プロトコル法と比較しながら説明しなさい。

| | ヒューリスティック評価法 | 発話プロトコル法 |
|---|---|---|
| 誰が評価するか | 専門家（評価者） | 実際のユーザ |
| 何を基準にするか | ニールセンの10ヒューリスティック | ユーザの自然な行動・発話 |
| コスト | 比較的低い・短時間 | 参加者確保・観察の負担が大きい |
| 何がわかるか | ガイドライン違反の有無 | 実ユーザがどこで詰まるか |
| 限界 | 実ユーザの反応は反映されない | 全思考が言語化されるわけではない |

どちらもユーザビリティ評価手法だが、ヒューリスティック評価法は専門家による基準照合であるのに対し、発話プロトコル法は実ユーザの思考プロセスを直接記録する点が異なる。前者はコストが低く早期発見に向き、後者は実際の使われ方を詳細に把握できる。

### 問題T
ミステークが起きやすい段階（計画・比較・評価）について、それぞれ具体的な例を挙げて説明しなさい。
計画 体調が悪いので風邪薬を飲もうと計画していたが実際食中毒だった
比較 睡眠薬飲む必要のあるユーザが飲んだことを忘れてしまってもう一度のでしまった
評価 コーヒーを飲むという計画を立ったがそもそも集中力が落ちているのは寝不足のせいだった


**計画（プラン）でのミステーク**
体調が悪い → 「市販の風邪薬を飲もう」と計画した → でも実際は食中毒だった
計画（判断）自体が間違っていた。正しい計画を実行しているのでユーザは気づきにくい。

**比較でのミステーク（記憶ラプス）**
睡眠薬を飲む習慣があるユーザが、すでに飲んだかどうか記憶があいまいなまま「まだ飲んでない」と判断して二度飲んでしまった
ゴールと現在の状態を照合する段階で記憶が抜け落ち、誤った比較結果を出してしまった。

**評価（ゴール）でのミステーク**
集中力が落ちている → 「コーヒーを飲もう」とゴールを立てた → 本当の原因は睡眠不足だった
そもそも立てたゴール自体が的外れだった。

---

### 問題U
スリップが起きやすい段階（詳細・実行・知覚・解釈）について、それぞれ具体的な例を挙げて説明しなさい。

詳細：細かい仕様がわからず違うことをやってしまった
実行：ボタンおそうとおもったら関係おないところを押してしまった
知覚：エアコンの音がならず近くできなかった
解釈：ダウンロードプログレスバーが止まっており処理中と解釈してしまいフリーズしてた


**詳細（詳細化）でのスリップ（記述類似性）**
メールを送ろうとして、宛先を「To」に入れると決めようとしたが「Cc」に入力する手順を組んでしまった
操作の手順を具体化する段階で、隣接する類似の対象に向かってしまった。

**実行でのスリップ（乗っ取り型）**
カップをテーブルに置こうとしたが、気づいたら冷蔵庫にしまっていた
詳細化まで正しかったが、身体的な実行段階で習慣的な動作に乗っ取られた。

**知覚でのスリップ（記憶ラプス）**
エアコンのリモコンを押したが、確認音が小さく気づかなかった → もう一度押してしまった
外界の変化（動作音・ランプ）を知覚できず、操作が正しく伝わったか確認できなかった。

**解釈でのスリップ**
ダウンロードのプログレスバーが止まっているのを見て「処理中」と解釈した → 実際はフリーズしていた
知覚はできていたが、その意味を誤って解釈してしまった。
